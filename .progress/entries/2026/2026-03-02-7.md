## Title
Align upstream flow with opencode Codex OAuth compatibility

## Background / Issue
OAuth login and callback succeeded, but `/v1/models` and `/v1/chat/completions` still failed with API-scope/quota errors because requests were sent to `api.openai.com/v1/*` instead of the ChatGPT Codex backend flow used by `opencode`.

## Actions / Outcome
- Approach 1: Probed real upstream responses with the acquired OAuth token and compared behavior against `opencode` plugin flow -> identified required Codex backend schema constraints (`instructions`, `input` list, `store=false`, `stream=true`).
- Approach 2: Implemented `codex_oauth` upstream mode with request/response translation: chat-completions -> Codex responses stream -> OpenAI chat-completions output; also added static compatibility model list for `/v1/models`.
- Final approach: Added mode/config defaults, account-id header extraction from JWT claims, and regression tests for codex compatibility path -> endpoints now return successful responses under OAuth login.

## Lessons / Refinements
- Successful OAuth callback does not imply compatibility with OpenAI API scopes; upstream endpoint semantics must match auth source.
- For Codex ChatGPT OAuth, protocol translation is required to preserve OpenAI-compatible downstream contract.
