## Title
Restore codex chat-compat tool-calling parity with responses translation

## Background / Issue
Users observed that OpenCode provider-catalog setup (`@ai-sdk/openai-compatible`) produced a different request/response shape versus direct codex OAuth behavior, especially missing `tool_calls` in chat-compat mode. Existing docs also stated tool fields were not translated.

## Actions / Outcome
- Approach 1: Reproduced mismatch through focused server tests for non-stream and stream chat paths with function tool calls -> confirmed that `tools`/`tool_choice`/`reasoning_effort`/tool-output history were dropped during chat-to-responses translation.
- Approach 2: Extended chat request conversion to map tool-related fields to responses payload (`tools`, normalized `tool_choice`, `parallel_tool_calls`, `reasoning.effort`, `max_output_tokens`) and convert message history (`assistant.tool_calls`, `tool_call_id`) into responses `function_call` / `function_call_output` items -> upstream now receives tool-call-complete context.
- Approach 3: Reworked SSE translation from codex responses back into chat outputs for both JSON and streamed chunk modes by handling `response.output_item.*` and `response.function_call_arguments.*` events -> gateway now emits chat `tool_calls` and uses `finish_reason: "tool_calls"` when function calls are produced.
- Final approach: Kept `/v1/responses` as raw pass-through for full codex event fidelity, while bringing `/v1/chat/completions` to full expressible parity for tool-calling workflows. Updated bilingual README/API docs to match behavior.

## Lessons / Refinements
- Chat-compat parity requires two-sided mapping: request translation is not enough; stream and non-stream response reconstruction must process function-call event families explicitly.
- Compatibility docs must track implementation closely; stale caveats can mislead debugging and mask fixed behavior.
