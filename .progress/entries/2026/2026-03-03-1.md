## Title
Log API prefix and discovered models on serve startup

## Background / Issue
Operators needed immediate runtime guidance after `serve` starts, including the base API prefix and a visible list of available models. The default sample config still pushed JSON logs, which reduced terminal readability for day-to-day local operation.

## Actions / Outcome
- Approach 1: Added startup probe helpers in `internal/cli/startup_probe.go` with TDD-first tests -> implemented API prefix rendering, wildcard host normalization, model discovery, retries, and warn-only fallback.
- Approach 2: Wired probe into `runServe` as a non-blocking goroutine after server launch -> startup now logs `api_prefix` and then logs discovered `available_models` (or warning on failure).
- Final approach: Aligned defaults and docs (`config.example.yaml`, README EN/ZH, OAuth setup EN/ZH) with human-readable text logging and startup metadata expectations -> behavior and onboarding guidance are consistent.

## Lessons / Refinements
- Separate display prefix and probe prefix handling to balance user-friendly output (`127.0.0.1`) with robust local probing (`localhost`) on wildcard listeners.
- Keep startup diagnostics best-effort and non-fatal so observability improvements never reduce service availability.
