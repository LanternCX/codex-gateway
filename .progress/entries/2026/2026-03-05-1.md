## Title
Normalize codex responses requests for OpenCode compatibility

## Background / Issue
OpenCode requests to `POST /v1/responses` in `codex_oauth` mode failed with `Bad Request: {"detail":"Unsupported parameter: max_output_tokens"}` and, in some cases, `Instructions are required` when upstream Codex backend validation rejected request fields.

## Actions / Outcome
- Approach 1: Added failing regression coverage for codex responses forwarding without `instructions` and with `max_output_tokens` -> reproduced upstream rejection path in `internal/server/responses_test.go`.
- Approach 2: Traced codex responses handler flow and confirmed request body was proxied with only JSON validation, without codex-specific normalization -> identified missing normalization boundary in `handleResponses`.
- Final approach: Added `normalizeCodexResponsesRequest` to inject default instructions and strip unsupported token-limit fields (`max_output_tokens`, `max_completion_tokens`) before codex upstream proxying, then updated bilingual API docs/OpenAPI to document normalization semantics.

## Lessons / Refinements
- Codex backend compatibility requires request-shape normalization on both chat-compat and responses pass-through paths, not only chat translation.
- Preserving client compatibility while avoiding upstream 400s is best handled at a single request-boundary normalizer in `internal/server`.
